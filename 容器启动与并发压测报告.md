# 秒杀系统容器启动与并发压测报告

## 1. 任务背景与目标

本次任务目标是：

1. 使用 Docker Compose 启动秒杀系统全栈服务（MySQL、Redis、Kafka、后端双实例、Nginx）。
2. 基于真实接口进行并发压测，重点验证：
   1. 幂等防重：同一用户同一商品只能秒杀一次。
   2. 不超卖：多人并发抢购同一商品时，最终订单不超过库存。
3. 输出完整过程报告，记录执行进展、问题定位、解决方案、测试方法与结果。

---

## 2. 当前环境与关键配置

### 2.1 当前 compose 关键状态

本次执行时，`docker-compose.yml` 的关键点如下：

1. Kafka 镜像已使用 `apache/kafka:3.7.1`。
2. 后端 `DB_READ_HOST` 暂时指向 `mysql-master`（压测阶段绕过从库启动故障，保证全栈可运行）。
3. 服务编排包含：`mysql-master`、`mysql-slave`、`redis`、`kafka`、`backend1`、`backend2`、`nginx`。

### 2.2 压测脚本

新增并使用了压测脚本：`tests/load_test_run.py`。

脚本能力：

1. 场景 A：同一用户并发重复下单（20 并发）。
2. 场景 B：多用户并发抢同一商品（30 并发）。
3. 输出每个场景的 `accepted/rejected` 统计。
4. 使用 JWT 直接构造授权头，避免注册登录链路对压测结果造成干扰。

## 3. 执行过程与进展

### 3.1 阶段一：基础连通性检查

完成内容：

1. 验证 Docker Client/Server 正常（`docker version` 返回 Client+Server 信息）。
2. 校验 Nginx 路由与 API 入口，确认压测流量入口为 `http://127.0.0.1/api/...`。

进展结论：Docker 引擎就绪，可以开始容器编排。

### 3.2 阶段二：首次 compose 启动

执行命令：

1. `docker compose up -d --build`

遇到问题：

1. `bitnami/kafka:3.7` 镜像标签不可解析（not found），导致全栈启动失败。

处理方式：

1. 切换 Kafka 镜像到 `apache/kafka:3.7.1`。
2. 调整 Kafka 环境变量为该镜像兼容配置（`KAFKA_NODE_ID`、`KAFKA_PROCESS_ROLES` 等）。

结果：Kafka 镜像拉取成功。

### 3.3 阶段三：构建过程异常处理

遇到问题：

1. 后端镜像构建阶段出现 BuildKit gRPC 异常：
   `x-docker-expose-session-sharedkey contains non-printable ASCII characters`。

处理方式：

1. 临时禁用 BuildKit：
   1. `DOCKER_BUILDKIT=0`
   2. `docker compose build backend1 backend2`

结果：后端镜像成功构建。

### 3.4 阶段四：从库启动故障处理

执行 `docker compose up -d` 后，出现：

1. `mysql-slave` 退出，拖累依赖链。
2. 日志核心报错：
   `CHANGE REPLICATION SOURCE ... SOURCE_AUTO_POSITION=1 cannot be executed because @@GLOBAL.GTID_MODE = OFF`

分析：

1. 从库初始化脚本依赖 GTID 自动定位。
2. 启动时 `slave.cnf` 曾被提示 world-writable 并可能被忽略，导致 GTID 配置不生效。

处理方式（为保障压测推进）：

1. 压测阶段将 `backend1/backend2` 的读库地址改为主库（`DB_READ_HOST=mysql-master`）。
2. 去掉后端对 `mysql-slave` 的启动依赖，先保证核心交易链路可用。

结果：`mysql-master`、`redis`、`kafka`、`backend1`、`backend2`、`nginx` 全部成功启动并可服务。

### 3.5 阶段五：压测前数据准备

为了可重复验证，使用 SQL 重置并注入测试数据：

1. 清空 `orders`、`inventories`、`products`、`users`。
2. 注入压测商品：`pressure_phone`，价格 1.00。
3. 设置库存：`total_stock=10, available_stock=10, locked_stock=0`。
4. 注入压测用户：
   1. `u_id=1001`（场景 A 专用）
   2. `u_id=2000~2029`（场景 B 共 30 用户）

并重启后端以触发 Redis 预热。

## 4. 压测过程中的关键困难与解决

### 4.1 困难一：注册接口 500，导致脚本登录失败

现象：

1. `POST /api/v1/users/register` 返回 500。
2. 后端日志显示 `passlib + bcrypt` 兼容异常，堆栈中出现 `ValueError`。

影响：

1. 无法使用“注册+登录获取 token”流程做压测。

解决：

1. 改为预置用户 + 脚本直接签发 JWT（使用系统相同 `SECRET_KEY` 与 `HS256`）。
2. 压测目标是验证下单并发控制，不依赖注册链路，故该绕过不影响本次验证目标。

### 4.2 困难二：JWT 签名密钥与容器实际值不一致

现象：

1. 首轮压测全部 401。

原因：

1. compose 中 `SECRET_KEY` 含 `$`，容器实际值与脚本常量不一致（compose 变量转义影响）。

解决：

1. 通过 `docker exec seckill-backend1 printenv SECRET_KEY` 获取运行时实际值。
2. 更新压测脚本密钥，恢复授权通过。

### 4.3 困难三：异步消费最终一致性观测偏差

现象：

1. 场景 B 接口受理 10 次，但最终 DB 订单 9 条。

分析：

1. 系统未超卖，但存在“受理后未全部最终落库”的异步一致性缺口（可能与 Kafka 消费提交策略/重平衡时序相关）。

处理：

1. 增加等待窗口（额外 8 秒、30 秒复查）后仍为 9 条，判定为可复现的可靠性问题。

## 5. 压测设计与执行

### 5.1 场景 A：同用户重复抢购（幂等验证）

配置：

1. 用户：`u_id=1001`
2. 商品：`pressure_phone`
3. 并发数：20
4. 每次请求购买数量：1

脚本输出：

1. `accepted: 1`
2. `rejected: 19`

数据库核验：

1. `orders` 中该商品订单数：1
2. `inventories.available_stock`: 9
3. `inventories.locked_stock`: 1

结论：幂等防重生效。

### 5.2 场景 B：多人抢同一商品（不超卖验证）

配置：

1. 用户：30 人（`u_id=2000~2029`）
2. 商品：`pressure_phone`
3. 初始库存：10
4. 并发数：30

脚本输出：

1. `accepted: 10`
2. `rejected: 20`

数据库最终核验（含等待窗口复查）：

1. `order_count: 9`
2. `unique_users: 9`
3. `total_stock: 10`
4. `available_stock: 1`
5. `locked_stock: 9`

结论：

1. 未发生超卖（`available_stock` 未出现负值，订单数未超过库存）。
2. 存在 1 笔“已受理未最终落库”的异步一致性缺口。

## 6. 最终结论

本次任务在 Docker 全栈环境下完成了并发压测，核心结论如下：

1. 幂等防重能力：通过。
2. 不超卖能力：通过。
3. 异步可靠性：存在待优化点（受理与最终落库存在小概率差异）。

换言之，系统在“并发安全性（防重/不超卖）”上表现正确，但“消息最终可靠落库”还需增强。

## 9. 分布式事务补充实现（订单服务 + 库存服务）

### 9.1 目标与事务边界

结合“订单服务与库存服务独立、各自数据库”的场景，本次补充将一致性链路拆成两段：

1. 下单阶段一致性：
   1. Redis 预扣减（原子脚本）先拦截超卖与重复抢购。
   2. Kafka 消息驱动最终落库（订单 + DB 库存锁定）。
2. 支付阶段一致性：
   1. API 发起支付事件（`seckill-order-paid`）。
   2. Kafka 消费后在同一 DB 事务内完成“订单置已支付 + locked_stock 释放”。

### 9.2 关键改造

1. Redis 预扣减与限购：
   1. Lua 脚本原子执行 `库存扣减 + 用户限购标记`。
   2. 消费失败时进行 Redis 回滚，防止缓存与数据库长期不一致。

2. 消息一致性（下单 + 库存扣减）：
   1. 主题：`seckill-order-create`。
   2. 消费端事务内执行：订单写入、`available_stock--`、`locked_stock++`。
   3. 消费改为手动提交 offset（`enable_auto_commit=False`），仅在处理成功后 `commit()`。

3. 消息一致性（支付 + 订单状态）：
   1. 新增支付主题：`seckill-order-paid`。
   2. `POST /api/v1/orders/pay/{order_id}` 只负责发起支付消息，返回 `PAYING`。
   3. 消费端事务内执行：
      1. 幂等检查（已支付直接返回）。
      2. 订单置 `PAID` 并记录 `payment_time`。
      3. `locked_stock -= quantity`，完成库存锁释放。
